# DESIGN-FEATURE-PAGE.md — 機能ページ デザイン仕様

**対象実装**: `app/<機能名>/page.tsx`（各機能の個別ページ）
**正準**: 機能ページの新規作成・修正はすべて本書に従う。カラー・タイポ・共通コンポーネントは `DESIGN.md` を参照。
**実装例**: `app/1to1/page.tsx`（1to1コミュニケーション機能ページ）

---

## 0. 機能ページの設計思想

### 目的

機能ページは「機能カタログ」ではなく、**お客さんが自分の課題を見つけるページ**。
「あ、これうちの困りごとだ」と止まるカードが並んでいる状態を作る。

### 核となる原則

1. **課題→解決のカード構成** — 機能名ではなく「お客さんの言葉で書いた課題」が主役。機能名は見出し、課題文がフック、解決文がクロージング
2. **グループで見つけやすくする** — カードを課題の性質でグルーピングし、見出しを目次として機能させる。14枚並ぶと探せないが、3グループに分かれていれば自分の関心領域が一目でわかる
3. **クロスセルを導線に組み込む** — 「他の機能と組み合わせる」セクションで、組み合わせた結果何が実現するかを具体的に書き、他の機能ページへリンクする
4. **アセットの成長がページの成長** — presales案件でベース化が決まった機能は、カードとしてこのページに追加される。pdm-digestでバックログに入ったものがそのままLPのカードになる
5. **開発前でも掲載する** — バックログにベース機能として入っていれば、実装前でもカードに出す。ただし「開発中」「未リリース」の表記は入れない
6. **デモがあるカードだけリンクを出す** — `demo_url` が設定されたカードのみ「デモを試す →」リンクを表示。全カードにデモがある必要はない

### DESIGN.md との関係

- カラー・タイポ・コンポーネント・CTA・フッターは `DESIGN.md` に従う
- 機能ページ固有のルール（ページ構成・カード仕様・グルーピング）は本書で定義

## 1. ページ構成

```
1. ヒーロー（ダーク背景）
2. 信頼バッジ帯
3. できること（グループ別カード一覧）  ← background: white
4. 他の機能と組み合わせる             ← background: muted
5. 導入ステップ                      ← background: white
6. CTA（ダーク背景）
7. フッター
```

### 背景色の交互配置ルール

隣接するセクションが同じ背景色にならないよう交互に配置する。

| セクション | 背景 |
|-----------|------|
| ヒーロー | dark（`#0a0a0a`） |
| バッジ帯 | white + border-bottom |
| できること | white |
| 組み合わせ | muted（`#F8F9FA`） |
| 導入ステップ | white |
| CTA | dark |
| フッター | dark（`#0a0a0a`） |

## 2. ヒーロー

DESIGN.md §7-1 に準拠。以下が機能ページ固有の要素:

### バッジ
```
<機能名>機能
```
LINE Green の pill バッジ。`bg-[#06C755]/20 border border-[#06C755]/50`

### 見出し（h1）
その機能の**コア体験を1文で**。機能説明ではなく、導入後に実現する状態を書く。

- OK: 「担当が替わっても、お客様との対話の歴史と好みが組織に残る。」
- NG: 「1to1チャット機能でスタッフと顧客がLINEで対話できます。」

LINE Green で強調する語は、コア体験のキーワード（2〜3語まで）。

### サブテキスト
機能の概要を1〜2文。丁寧体。

### CTA
- 左: 「無料で相談する」（primary, お問い合わせフォームへ）
- 右: 「デモを試す」（outline white, デモURLへ）※デモがない機能は省略

### チェックリスト
課題ベースの言葉を3つ。技術用語ではなく、お客さんが共感する表現にする。

- OK: 「個人LINE脱却」「担当引き継ぎ」「24時間対応」
- NG: 「対話履歴の永続保存」「AI自動要約」「MA連携」

## 3. できること（グループ別カード一覧）

### グルーピング

カードを**課題の性質**でグルーピングする。グループ数は2〜4が目安。

各グループに:
- **ラベル**（h3, `text-xl sm:text-2xl font-bold`）— そのグループが何を解決するか
- **サブタイトル**（`text-sm text-[#6B7280]`）— 1行の補足

グループの分け方の例（機能によって異なる）:
- 1to1: 「基本のチャット機能」「自動化」「データ活用」
- 会員証: 「登録・発行」「データ管理」「連携・活用」
- 予約: 「予約受付」「リマインド・変更」「分析・最適化」

### カード仕様

```
┌────────────────────────────────┐
│ [アイコン] 機能名                │  ← h3, font-bold
│                                  │
│ 「課題文」                       │  ← text-sm text-[#9CA3AF], 「」付き
│                                  │
│ 解決文                           │  ← text-sm text-[#4B5563]
│                                  │
│ デモを試す →                     │  ← demo_url がある場合のみ
└────────────────────────────────┘
```

#### アイコン
- Lucide アイコンを使用
- `w-10 h-10 rounded-lg bg-[#E8F8F0]` の背景円に `w-5 h-5 text-[#05A847]` のアイコン

#### 機能名（20字以内）
その小機能を端的に表す名前。

#### 課題文（重要）
**お客さんの言葉**で書く。現場の具体シーンが浮かぶ表現にする。

- OK: 「営業時間・アクセス・返品ポリシーなど、同じ質問にスタッフが何度も答えている」
- NG: 「同じ質問に何度も答えている」（一般的すぎて自分ごと化しにくい）
- OK: 「「予約の変更は？」「修理の流れは？」など定型の問い合わせにもスタッフが毎回対応している」
- NG: 「定型的な問い合わせにもスタッフが毎回対応している」（抽象的）

#### 解決文（1〜2行）
グロースパックでどう解決するかを簡潔に。機能の動作を書く。

#### デモリンク
`demo_url` プロパティが設定されたカードのみ表示。
`text-sm font-semibold text-[#05A847]` + ArrowRight アイコン。

### グリッドレイアウト

```
lg: 3列 (lg:grid-cols-3)
sm: 2列 (sm:grid-cols-2)
mobile: 1列
gap: gap-4 md:gap-6
```

グループ間のスペース: `space-y-12 md:space-y-16`

## 4. 他の機能と組み合わせる

### 目的

クロスセル。1to1を見に来た人に「会員証と組み合わせるとこうなる」を示して、他の機能ページに送る。

### カード仕様

```
┌─────────────────────────────────┐
│ [アイコン] <この機能> × <連携先>    │  ← h3, font-bold
│                                     │
│ 組み合わせた結果、何が実現するか    │  ← text-sm, 2〜3行
│ を具体的なシーンで書く              │
│                                     │
│ 詳しく見る →                        │  ← 連携先の機能ページへリンク
└─────────────────────────────────┘
```

#### 見出しの書き方

「× 会員証」ではなく「1to1 × 会員証」のように**このページの機能名を左**に置く。

#### 説明文の書き方

「連携できます」ではなく、**組み合わせた結果の体験**を書く。

- OK: 「チャット開始時に会員ランク・購買履歴・前回来店日を自動表示。スタッフは「初めまして」のやり取りなしに、お客様の文脈を把握した状態で対話を始められます。」
- NG: 「会員証機能と連携して、顧客プロフィールを参照できます。」

#### リンク先

連携先の機能ページが存在する場合: `/memberscard` 等へリンク。
存在しない場合: `/#features`（トップの機能一覧）へ暫定リンク。機能ページ作成後に差し替え。

### グリッドレイアウト

```
sm: 2列 (sm:grid-cols-2)
mobile: 1列
gap: gap-4 md:gap-6
```

Card variant: `elevated`, padding: `lg`, rounded: `xl`

## 5. 導入ステップ

3〜4ステップ。機能ごとに内容は異なるが、構成は共通。

```
[数字] Step N
タイトル
説明（1〜2行）
```

`lg:grid-cols-4`（4ステップの場合）、数字は `bg-[#05A847]` の円。

## 6. SEO・メタデータ

### layout.tsx

各機能ページに `layout.tsx` を作成し、以下を設定:

- **title**: `<機能名>｜<コア体験を1文で>｜グロースパック for LINE`
- **description**: ヒーローのサブテキストと同等
- **keywords**: 機能名 + 各カードに関連するキーワード（チャットボット、FAQ、時間外対応 等）
- **canonical**: `https://lp.growthpackforline.classmethod.net/<パス>`
- **OGP**: title/description を転記

### JSON-LD

- `Service` スキーマ（serviceType, name, description, provider）
- `BreadcrumbList`（トップ → 機能ページ）

## 7. データ構造

### FEATURE_GROUPS

```typescript
const FEATURE_GROUPS = [
  {
    label: string;       // グループ名（h3見出し）
    subtitle: string;    // 1行の補足説明
    cards: [
      {
        icon: LucideIcon;   // Lucide アイコンコンポーネント
        name: string;       // 機能名（20字以内）
        challenge: string;  // 課題文（お客さんの言葉）
        solution: string;   // 解決文（1〜2行）
        demo_url?: string;  // デモURL（任意）
      }
    ];
  }
];
```

### COMBINATIONS

```typescript
const COMBINATIONS = [
  {
    icon: LucideIcon;     // 連携先機能のアイコン
    name: string;         // 「<この機能> × <連携先>」
    description: string;  // 組み合わせで実現する体験（2〜3行）
    href: string;         // 連携先機能ページのパス
  }
];
```

### STEPS

```typescript
const STEPS = [
  {
    step: string;         // 「Step 1」
    title: string;        // ステップ名
    description: string;  // 説明（1〜2行）
  }
];
```

## 8. カードの追加フロー（PdMパイプライン連携）

機能ページのカードは、PdMパイプラインの出力として追加される。

```
1. presales案件で新しい課題が見えた
2. pdm-digest でアセット化判定 → ベース機能としてバックログに追加
3. pdm-publish で機能ページ用カードデータを生成
   - 機能名（20字以内）
   - 課題文（お客さんの言葉、現場シーンが浮かぶ表現）
   - 解決文（1〜2行）
4. FEATURE_GROUPS の該当グループにカードを追加
5. デモが作れるレベルまで仕様が固まったら demo_url を追加
```

### グルーピングの見直し

カード数が増えてグループ内が6枚を超えたら、グループの分割を検討する。
1グループ3〜6枚が見やすさの目安。
